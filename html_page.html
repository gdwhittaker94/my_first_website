About HTML

HTML means "HyperText Markup Language" and it was invented in 1989 by Sir Tim Berners-Lee. Don't worry, that also meant nothing to me when I started. That is why there is a picture of a skeleton on the left. You can imagine building a web page like the human body.

First, before anything else and arguably the most essential part, we need a structure on which we hang and hold everything. For a human that part is the skeleton. In the web development world, that structure is HTML.

HTML is a coding language that 'marks up' (converts) regular text and media content into code so that it can be displayed correctly by a computer (web browser) on a web page. Additionally, HTML allows for 'hypertext'. That means, ordinary text that can contain a 'hyperlink' (just 'links' to you and me) which allows you to go off and visit another website somewhere on the internet - arguably one of the best things about the internet! Altogether that gives us the name: HyperText Markup Language.

It is a coding language which instructs the web browser on where to put headings, paragraphs, images and more. HTML is all about structure. We never see any of it on our web page, but it's always there in the background invisibly structuring and ordering everything so it appears as we want it to. It has nothing to do with beautifying the web page or making it interactive and functional, for that we need other coding languages. A website built just with HTML looks something like this.

So HTML is the basis for every single web page we see on the Internet, and therefore an essential programming language to learn. Below are examples of the different parts of HTML code, what it looks like and what effect it has on a web page.

HTML Core Components & Ideas

Fundamental concepts to understand before starting to code with HTML

Tags and Elements

A tag looks like < this > and contains a keyword such as "h1".

Tags, keywords and elements are the fundamental building blocks of any HTML document.

Tags are the units of angle brackets and keywords, which go inside the brackets. There are opening tags "<...>" and closing tags which come afterward the content and have a forward slash in them "< /...>."

Keywords are words, letters or a combination or letters and symbols that have special effects within HTML and the content which they apply to. For example "h1" is one of these elements and it means 'Heading 1'. Using this tag defines an important heading within the HTML document and turns regular-sized text into a big, bold heading text. We will see the "h1" effect later.

Content is what we put between these tags and the elements. For example: < h1 > My heading < /h1 >

An element refers to the combined total of the tags (angled brackets and keywords) and content altogether.

Elements, combining all these different parts together, form a block or a unit within our HTML page. They now become their own entity within the document, fleshing it out and giving it substance.

For example: "< h1 > Example < / h1 >"

Extra Notes

It's not always necessary for both the opening and closing tag. Sometimes tags are 'self-closing' or 'empty'. For example: "< br >" means 'break' and causes a line break in your text. It doesn't need a closing tag to cause its desired effect.

Elements can go inside other elements. These are called nested/child elements and this means you can cause multiple effects upon a single piece of content. Try to close the elements in the same order they were opened. For example: "< p > Here is some < b > < u > bold, underlined < / b > < / u > text. < / p >, which looks like this:

HTML Attributes

Attributes are secondary keywords that go inside the opening tag of HTML elements. They are add-ons to these first-level HTML keywords mentioned before and allow us to play with them.

It's similar to how in video games we can add modifications and upgrades to certain objects such as weapons in order to change their appearance, their function, to enhance them etc. HTML attributes allow us to modify or 'upgrade' our HTML elements, to go beyond the basic default settings and customise our web page how we want or need. The same is true in the video game; without weapon add-ons and upgrades we would just be stuck with the same basic weapon the game developers gave us at the start of the game, which might be limiting or boring.

They normally consist of attribute="value" pairs where the value is wrapped in double quotation marks.

For example, the attribute 'title' is another keyword in HTML, but its importance is secondary in comparison to the keyword 'h1', for example. That is because 'title' always comes, and only can come, after a primary keyword - in this case, 'h1'.

The 'title' attribute has the effect of giving the the viewer extra information when they hover over something on the web page (aka a tooltip). If we don't include it, we just have the original element as it's programmed to be on our web page. Look at the example below:

What the code looks like: ... < h1 title="United States of America" > USA < /h1> ...

The effect. Hover over this text: USA

And now the original tag/element without the title attribute applied to it. Notice now that nothing happens: USA

Extra Notes

It might seems obvious, but the '=' in the attribute="value" pairs means "The value of X is equal to ...". So in the above example the code is saying "The value of title is equal to United States of America".

You are not limited to adding just one attribute to a tag and its keyword. You can add as many as you like! For example: < h1 title="United States of America" style= "color:red" > USA < / h1 >. Now look at the result: USA

Some attributes are actually required for certain primary keywords to work! For example the < img > tag requires the attributes 'src' meaning "source" - where the image is coming from - and 'alt' meaning "alternative" - a short text summary of the image which displays if the image can't load for some reason.

Lastly, not all attributes work with all keywords. You can investigate this more in the link previously given or here it is looked at in detail also.

HTML DOM (Document Object Model) or HTML Document Tree

Each HTML document can actually be referred to as a DOM (document object model) or simply a Document Tree - there is only a technical difference between the two terms (see here). Here we are specifically talking about the relationship between each element, describing the relationships between elements like we would describe a family tree. There are ancestors, descendants, parents, children and siblings. It is important to understand the DOM tree for later when we want to use other coding or programming langauges like CSS and Javascript.

A descendant refers to any element that is connected to, or contained by, a higher level element in the document tree - no matter how many levels lower. In the picture we can see that all elements that are connected below the < div > element are descendants of that < div >.

Extra Notes

The < html > tag in a document is also known as the root element. It is connected to, or contains, all the other elements and therefore is the sole parent element to all the other elements.

In fact, technically, everything in a HTML document is also known as a node. The entire HTML document is also known as the 'document node', the document tree the 'node tree', every HTML element an 'element node', the text inside HTML elements are 'text nodes', every HTML attribute an 'attribute node', all HTML comments are 'comment nodes' and so on.

That is, everything in our HTML document can be split up and categorised into its individual parts and be called a 'node'. This is useful later when we use CSS or Javascript on our website and we want to be able to access or target specific parts of a our website in order to modify or alter them.

Hyperlinks (Links)

Links connect one web page or website to another one. For example, this link will take you back to my website's home page.

They require two parts:
1) the "< a >" tag
2) the href= attribute.

Links are what make the web so special. Anything on the web can be connected to via a link: an image, an audio or video clip, a PDF file, an HTML document or an element within the document itself. It is the ability to go from one web resource to another on any server in the world, at the speed of light!

A link, in order to work, requires two parts. The "< a >" or anchor tag which acts as the container for the link destination and description, and the "href" or hyperlink reference attribute which is the web page you are directed to. The content part is what the viewer sees.

For example, if I wanted to redirect you to Google's search page, I would have to write something like this: < a href="https://www.google.com/"> Google < /a >, which looks to you like this: Google

When you link to a web resource, you have two options:

1) A local resource located in the local web page file (for example, on your computer).
2) A resource on the web located on a different web server.

If I send you back to the home page, for example, that HTML document is located within the website's folder, so you don't need to be sent to a different web server in order to get there. The link in the HTML document will simply read href=index.html.

But if I send you to Google's search page, that web page isn't located here within this website or the local folder, it's located somewhere on the web, so the link has to include the http://www. part and looks like the example I wrote earlier.

Images

The self-closing < img > tag allows us to add images to our web page and enhance its visual appearance. Two attributes are required to add an image:

1) the src= attribute (source) which tells the browser where to find the image.

2) the alt= attribute (alternative) which is a short text summary of the image which displays if the image can't load for some reason.

To add an image to our website, as well as the tag and attributes above, it's important in the src= attribute to correctly tell the browser where to get the image from. If the image is on our computer and stored in a local folder, then we must include the folder name in the value of the src= attribute. That is, the folder pathway to the image. Or, if we are sourcing our image from the web, then we need to include an url (starting with www. etc). Compare the source code of the two examples below:

Source code:
< img src="./images/googlelogo.png" alt="Google Logo" width="200px" height="70px" style="padding-left: 5vw;" >


Source code:
< img src="https://www.google.com/./images/branding/googlelogo/2x/googlelogo_color_272x92dp.png" alt="Google Logo" style="max-width:200px; padding-left: 5vw;" >

Additionally, sizing images using the < img > height= and width= attributes is important to do because usually the image at first does not appear on the screen as we intend it to. So we have to control it and manipulate it using these attributes in order to do this.

The < img > height= attribute tells the browser how high the image should measure and the < img> width= attribute tells the browser how wide the image should be. These attributes require numerical measurements in pixels (e.g. 400px) in order to work.

Alternatively, we can use the CSS style= attribute and the 'height:' and 'width:' properties which allow more measurements types than just pixels (e.g. 40%, 60vw, 80vh, 2em etc).

However, these measurements give the image a fixed size! If we want an image to be responsive and adapt to different screen sizes and types, then we need to use the CSS 'width:' and 'height:' properties again. Here, we set the width="100%" (or less if we want it to) and the height:"auto". This means the picture will automatically scale up and down to the screen size. This is exactly what I've done with the image below, and if you change the window size you will notice it adapts to the change:


Source code:
< img src="./images/googlelogo.png" alt="Google Logo" style="width:80%; height: auto; padding left: 5vw" >

However, sizing the image this way means it can be scaled up to be larger than its original size! So then, a better solution, in many cases, is to use the 'max-width:' property and a specific size measurement (px, vw, vh, % etc) instead via the CSS style= attribute. This keeps the image under control as it cannot exceed a certain size limit and so we can always be sure of how it will look on the screen, but it can still adjust to smallar screen sizes. This is the property I used on the second Google logo image from earlier.

Extra Notes

Above we looked a little bit at image responsiveness and how to make an image adapt to the size of different screens. Another way of making our web page more responsive is by using the < picture > tag. This tag allows the web developer to define multiple versions of an image to target different types of user devices and their subsequent different screen sizes. The user's device will then decide which image to best display on its screen. The two main reasons this tag exists are:

1) Small screens don't need large image files. They don't reap any benefit, but it takes just as long to download them.

2) Not all browsers or devices support all image formats. The picture element allows you to add images of all formats (e.g., .png, .jpg, .gif etc) and the browser uses the first one it recognises, ignoring the rest.

In order to do this you write the opening < picture > tag and between this and the closing tag you write the < source > tag. In the source tag you add the media= attribute (which defines when each image is the most suitable, under what screen conditions) and a srcset= attribute which refers to the image being targeted when said conditions are met. You should include around three to four different image versions/sizes and therefore three to four < source > tags.

Lastly, you specify an < img > element with the src= attribute which acts as a last resort if the < picture > tag is not supported by the browser or if none of the < source > tags match.

Class & id Attributes

The class= and id= attributes allow us to target specific HTML elements within our web page in order to give them certain styling (via CSS) or to alter them in some way (via Javascript).

If we want to style the HTML tags/elements on our page via an internal or external CSS stylesheet we have three different ways we can do this:

Method #1: Write the HTML tag/element keyword and give it some styling.

For example, in our internal or external CSS stylesheet we write: "h1 {color:green; padding:10px}". However, the negative aspect of styling this way is that the style rules will be applied to all the h1 tags/elements on our website - every single one! This is frustrating and perhaps limiting, especially if we want different header stylings across our website. So how can we style individual tags/elements?

Method #2: Apply the class= attribute to the individual HTML tag/element we want to style.

The class= attribute can be applied to any HTML element. For example, in our HTML document we have a < h1 > element and we want to style it. So inside the tag we write the class= attribute and give it a value. This value is essentially a name or label which acts as a bridge between the HTML element and the CSS styling intended for it, allowing the styling rules to 'cross' the bridge and take effect. Remember, HTML and CSS are two different coding languages and so need to be connected somehow in order for the CSS to work with HTML. So, let's see how above theory looks like in practice:

< h1 class="header1"> ...< /h1 >

Then, in our internal or external CSS stylesheet, we put a dot (.), aka the 'class selector', and the value (name/label) we gave our class attribute in the HTML element, followed by curly brackets '{...}'. So this above theory in practice looks like this:

.header1 {...}

Inside the curly brackets '{...}'' go the CSS styling rules which will be immediately applied to the < h1 > element. For example, color: blue; font-size: larger; etc...

There are some important things to know about the class= attribute:

  • It's possible to create multiple classes (multiple class= attribute values) within the class attribute. That is, one HTML element can belong to several classes.

    For example, a class attribute with the value of "index-h1" or "index_h1" is just one class name (due to the hyphen and underscore lines joining the words together) and would be targeted in CSS like this '.index-h1'.  However, "index h1" are two separate class names and could be targeted separately in CSS like this '.index {...}' and '.h1 {...}'. We might do this in order to separate our styling rules, which we might do because...


  • It's possible to apply the same class attribute value across multiple elements, and thus apply the same styling rules to those elements.

    For example, if we have two < h1 > elements on our HTML document, and we give them both a class attribute with the value "index-h1" (< h1 class="index-h1" >), then both these < h1 > elements will receive the exact same styling rules and will look the same on our web page.

    If we have two < h1 > elements on our document and we want some but not all of the styling rules from the first one to apply to the second one, if we separated the class values ("index h1" not "index-h1") then we can give our second element a class attribute value of "index" and all the '.index' styling rules will be applied, but then create another class attribute value (say "h1_2") with distinct styling rules so our second < h1 > element looks different from the first.

    Furthermore, because we can apply distinct class attribute values to distinct HTML elements, we could also apply our ".index" styling rules intended for the Header 1 elements to a totally different element like a < p > element as well.

  • It's possible to target elements inside a parent element with the class= attribute.

    For example, we have a < div > element (a container element) and inside this container we have a few paragraphs using the < p > tag. The < div > element is the 'parent' element and the < p > elements are the 'child' elements. Instead of adding a class attribute to every single paragraph tag we can just have one class in the < div > tag (e.g. class="indexIntroDiv"). Then, in our CSS stylesheet we write the name of the class as usual, .indexIntroDiv, but then followed by the child element inside the parent element we want to target. So our CSS code will look something like this:

    .indexIntroDiv p {...} ... which is like writing this ... .indexIntroDiv --> p {...}

    And this works because in HTML there is a hierarchical structure. That is, if a parent element is affected or altered in some way, then all its children elements are effected too. For example, in the HTML document the < body > tag is at the top of our visible web page hierarchy. If we add styling to that element then all elements below it will receive that styling too. That is to say, our entire web page!

  • Class= attribute values are case sensitive.

    That is, giving one tag a class= attribute value of 'Index-h1' is not the same as giving another tag a class= attribute value of 'index-h1'. If we forget this often this can cause confusion when we don't get an expected styling result on our web page.

Method #3: Apply the id= attribute to the individual HTML tag/element we want to style.

The id= attribute can, like the class=, be applied to any HTML element and works very much in the same way as the class= attribute. For example, repeating the example from before but using the id= attribute our example now looks like this:

< h1 id="header1" > ...

This time, in our internal or external CSS stylesheet, we put a hastag symbol (#) aka the 'id selector' and the value (name/label) we gave our id attribute in the HTML element, followed by curly brackets '{...}'. So our CSS code will look something like this:

#header1 {...}

Again inside the curly brackets '{...}' go the CSS styling rules which will be immediately applied to only to this < h1 > element. For example; {color: blue; font-size: larger;} etc...

There are some important things to know about the id= attribute:

  • The id= attribute's value is unique among all the id= attributes on the web page and cannot be repeated or reused.

    Unlike the class= attribute whose value can be applied to multiple HTML tags/elements, the value of the id= attribute can only be applied to one HTML tag/element. So in our above example, our CSS styling rules will onlybe applied to the HTML element with the id="header1", we cannot have a second < h1 > element and also apply the id="header1" to it. Instead, we would have to create a different id value such as id="header2". So this means that any CSS styling rules we apply to the an element through the id selector and id attribute will always be unique to that element within our HTML document.

  • You can only create one id= attribute value per HTML tag/element.

    Unlike the class= attribute where you can create several class values within the attribute, there can only be one value within the id= attribute.

  • The id= attribute value allows you to make bookmark anchor links within your website.

    This is the main reason we have the id= attribute! This attribute allows us to create links the move the user around the current web page or website. More detail about this can be found here.

Layout Tags: Semantic and Non-Semantic Container Elements

Layout tags/elements are fundamental pieces in our HTML code which help us structure our web pages.

Each tag we employ in our HTML code creates a virtual container on our web page which we can put stuff inside of (content, other HTML elements, pretty much anything!). The container keeps that stuff safely grouped together and separated from the other stuff on our web page. In essence, we use these virtual containers to structure and divide our web page up, keeping certain content and elements together while simultaneously separating it from other content and elements. If you look at any average web page on the internet you will see how it is divided up into boxes and sections. This is because of the layout tags/elements.

The difference between the semantic and non-semantic container tags/elements is that the semantic tags have a clear meaning or purpose, we know more or less what kind of content will be inside them. The non-semantic tags don't have a clear meaning and can contain any type of content.

At first making nested divs can be very confusing. The key is to make sure you line up the < div > tags on the left correctly, creating a 'sandwich' effect. That is, you have the first top and bottom layer (imagine, the top and bottom layer of bread of your sandwich) within which you can put some filling (e.g. some text or 'lettuce'). Then within that same layer you have another top and bottom div layer (another two slices of bread) within which you can put some independent filling (e.g. an image or 'cheese'). Then again within this second sandwich layer you can put another top and bottom div layer (another two slices of bread) within which you can put some different filling (e.g. an iframe or some 'ham').

< span > tag
This tag is used for containing text more than anything else. It allows us to separate some words or a sentence or so from the rest of a text. From there, we can style these words via CSS or target them via Javascript. Again, just placing the < span > in our code doesn't actually do anything to the visual appearance of our page, we have to style it or its content via CSS to see a difference.

This is an example of some text within a paragraph element that has a span tag inside it, allowing the words within the span tags to be different from the rest of the text.

By default the < span > container is an inline-level element: it will appear on the same line as it is created on without causing any line breaks before and after it.

SEMANTIC

Note: By default all these tags/elements are block-level tags.

Below you can see some images which give you an idea of how the different following semantic elements can be used to clearly and logically structure our web pages both for us and for others. That is, instead of just using a load of meaningless 'div' containers:

< header > tag
This container is used for introductory content or a set of navigational links. Because of its name, we already know more or less what to expect inside it and where we will find it on our web page or in our HTML code. Typically the < header > container contains things like:

  • Heading elements e.g. < h1 > ... < h2 >
  • a logo or icon
  • author information
  • a search bar
  • navigation links

Note: It is possible to have more than one < header > element on our web page, however, this tag cannot be placed within another semantic element.

< nav > tag
This container is used for containing navigation links, hence the 'nav' name. It can contain navigation menus, quicklinks, table of contents etc. However, not every link on the website should be included here, just the most important ones.

Note: It is possible to have more than one < nav > element on our web page.

< main > tag
This container is used for the primary or most important content of a web page. In this container is the core topic of the web page. Its content should be unique to this web page and shouldn't contain any content that is repeated across other web pages such as sidebars, navigation links, copyright information, logos, and search forms.

Note: It is not possible to have more than one < main > element on our web page. Also, the < main > tag cannot be a descendant of any of these semantic layout tags: < article >, < aside >, < footer >, < header >, < nav >.

< section > tag
This container is used for defining the area of a stand-alone section of our web page. That is, a collection of content which should be kept together and separate from other web page content. Usually the section is topped with a heading at the start of it and can be used for things like:

  • Chapters
  • Introductions
  • News Items
  • Contact Information

Of all the semantic tags, < section > is the closest to a < div > tag because the name in itself doesn't cleary define what sort of content it contains.

< article > tag
This container is used for independent, normally self-contained text content that should make sense on its own or when placed in a context outside the website. This container can be used for things like:

  • Forum posts
  • Blog posts
  • User Comments
  • Newspaper articles
  • Product Descriptions/Cards

The < section > vs < article > tag
These containers seem similar and indeed they are. They both define a collection of content that makes sense by itself and for that reason is separated from other web page content. In general the < section > tag defines spacially (i.e. this area is a distinct section) whereas the < article > tag defines textually (i.e. this text is distinct from other sections of text). However, at times you will find < article > elements containing < section > elements and vice versa.

< aside >
This container is used for content which is related to the page's main content but is not the primary page content; it's not the most important content. This container can be used for things like:

  • Sidebar
  • Call-out boxes
  • Advertisements
  • Product Shipping Information

< footer > tag
This container is used for summary information or concluding content of a web page. It can be used for things like:

  • Authorship Information
  • Copyright Information
  • Contact Information
  • Sitemap
  • Back to top links
  • Related Documents

Note: It's possible to have more than one < footer > element on our web page.

The HTML < head > element

The < head > element of an HTML document is a container for metadata (data about data, information about the document) which is not seen by the user because it is placed after the < html > tag at the start of the document but before the < body > tag where all visible web page content begins. This data is vital for our HTML document in order for it to work and display correctly in the web browser. It contains other important HTML tags/elements such as:

  • The < title > tag
  • The < meta > tag
  • The < style > tag
  • The < link > tag
  • The < script > and < no script > tags
  • The < base > tag

Note: these tags must go in the < head > section of our document and not anywhere else. Below is an examination of each of these tags or elements and their function within the < head > element:

< title > tag
This tag is obligatory and its content becomes the name of the web page. That is, it's what you see in the browser's title bar/page tab at the top of the screen and is how the page is named if you save it to your favourites tab. For example, the name of this web page is 'HTML' (look at this window's tab at the top of your screen). The name we give our web page is important in order to optimise search engine optimisation. < meta > tag
This tag gives us metadata about the HTML document (data about data, information about the document). Some of which provides data about our web page, some of which tells the browser how to display the content of the document. We can use as many < meta > tags as we want in the < head > section, all of which are useful for browsers, search engines and other web services. The metadata we can define in the < meta > tag includes:

  • A Page Description
  • Page Keywords
  • The Author of the Document
  • Viewport Size Settings
  • HTML Character Set

We can add this kind of metadata to our < meta > tag by applying two obligatory attributes to the tag:

  1. name= attribute: the value of this attribute tells the browser and search engines what type of metadata we are listing here in the secondary content= attribute. Here we can specify one of many different values:

    • "application-name": specifies the name of the web application that the page represents

    • "author": specifies the name of the author of the document. The author can be an individual, the company as a whole or a third party. For example: name="author" content="John Smith"

    • description: provides a short synopsis of the page that search engines can pick up and use when the page web page appears in search results. We saw this in the image above in the < title > tag section. For example: name="description" content="Free Web Tutorials"

    • "keywords": specifies a comma-separated list of keywords, relevant to the page. These tell search engines what the page is about and give extra weight to a document's keywords. It's good practice to always specify keywords in our metadata set. For example: name="keywords" content="HTML, meta tag, tag reference"

    • "viewport": the viewport is the visible area of a web page that the user sees and is not the same on a mobile phone as it is on a computer. For that reason it's important to tell the browser how to control the page's dimensions and scaling via this < meta > tag so that mobile viewers get an equally good viewing experience of our website. Without a viewport meta tag, mobile browsers render the web pages at typical desktop screen widths, and then scale it down to fit the mobile screen. As a result, it requires pinch-and-zoom to view the web page properly in mobile devices, which is very inconvenient.

      We typically give the content= attribute a value of: "width=device-width, (sets the width of the viewport to the same as the screen width of the device) and then: ..., initial-scale=1.0" (which sets the initial scale or zoom level to 100% when the page is first loaded by the browser).

      So our < meta > tag will look like this:

      < meta name="viewport" content="width=device-width, initial-scale=1.0" >

  3. charset= attribute: This is not an obligatory attribute, but another (and the only other) other attribute we might apply to our < meta > tag. This attribute can apply to the < meta > tag on its own (without the above two attributes) and tells the browser the character encoding for our HTML document.     < style > tag
    This tag is used to define CSS style rules within the HTML document. These rules change the look and appearance of our web page completely.

    In CSS we have different ways to style our elements:

    1. Inline Styling, using the style= attribute: styling only one, individual element by placing this attribute within the opening tag of the element in our HTML document. This way of styling is of a high-specificity level because it takes precedence over the following methods two and three of this list and so should only be used if we want a specific tag/element to be excluded from more general styling rules. However, it has the limitation of not being able to offer all the features that come with those methods like pseudo elements, pseudo classes and media queries. Also, it makes our code look messier and harder to read.

    2. Internal Stylesheet using the < style > tag: a collection of styling rules within the HTML document, normally connected to our HTML document elements/tags via class= attributes and id= attributes. This way of styling is of a medium-specificity level because the styling rules can apply to the whole document (but not outside it). Therefore, we should only really use this form of CSS styling if we have a single HTML document with a unique style. If we repeat the same styling rules across several documents, it's better to use the last option...

    3. External Stylesheet using a separate '.css' file: a collection of styling rules outside the HTML document, connected to our HTML document elements/tags via class= attributes and id= attributes. This way of styling is of a lower-specificity level because the styling rules can apply to the whole website. This is the default way to style our HTML document using CSS.


    < link > tag
    This tag is used to establish and define the relationship between the current HTML document and an external document or resource. It loads a linked resource into our HTML document. A common example is using this tag to connect our HTML document to an external CSS stylesheet or to bring a favicon into our web page page like I demonstrated here. It is a self-closing tag and only contains attributes. An example: < link rel="stylesheet" href="mystyle.css" >

    Here is a list of possible attributes we can apply to the < link > tag:

    1. rel=: this attribute is always required and specifies the relationship between the current document and an external source. The values of this attribute are listed and explained in detail here and here. For example: < link rel="stylesheet" href="styles.css" >

    2. href=: we give this attribute an URL value which specifies the location of the linked document. For example:

      < link rel="stylesheet" href="styles.css" >

    3. type=: specifies the media type(the format and nature) of the linked href= file/resource. Browsers don't look at a link's file extensions, but rather what media type it is. Typically we just link a stylesheet to our HTMl document whose type= value is "text/css". However, since CSS is the only stylesheet language, if we omit the type= attribute the browser will look at the rel=attribute to guess the correct type. So, if rel="stylesheet" is present, the browser will assume the type is "text/css". An example:

      < link rel="stylesheet" type="text/css" href="styles.css" >

    4. hreflang=: specifies the language of the text in the linked document.This attribute is only used if the href attribute is set. It's mainly useful for search engines. Its value can contain four characters: the first two represent the language code and, if specified, the last two represent the country code. For example, the hreflang value of 'en-GB' specifies the English language in Great Britain. Another example:

      < link rel="preload" as="video" type="video/mp4" href="/media/movie.mp4" hreflang="en" >

    5. media=: specifies what media or device the target resource is optimised for and is mostly used with CSS style sheets to specify different styles or web page appearances for different media types. The default value is 'screen' (computer screens, tablets, smart-phones etc.), but other values include 'all' (all media type devices), 'print' (print preview mode/printed pages), and 'speech' (screenreaders that "reads" the page out loud). More than one value can be given (each separated with a comma). For example:

      < link rel="stylesheet" type="text/css" href="print.css" media="print, speech" >

    6. sizes=: this attribute is only used if we have a rel="icon" attribute before it. It specifies the size (height and width) of the linked icon/favicon resource. This size is written in this order (height first, width second), separated by an 'x'. For example:

      < link rel="icon" href="/img/html.png" type="image/png" sizes="32x32" >


    < script > tag
    This tag allows us to add/embed executable code or 'script' to our web page, usually Javascript, which is processed on the user's computer and makes the web page interactive. The tag and code are invisble to the user. The code can be declared in-line, inside the < script > tag, as many times as we want in our < head > or < body > sections, or we can load external script files containing all the excutable code using the src= attribute. For example:

    < script >
       document.getElementById("demo").innerHTML = "Hello JavaScript!";
    < /script >

    The < no script > tag is used to provide an alternate content for users that have disabled executable code/ scripts in their browser or the browser doesn't recognize the scripting language, or have a browser that doesn't support client-side scripting. For example:

    < script >
       document.write("Hello World!")
    < / script >
    < noscript > Your browser does not support JavaScript! < /noscript >

    < base > tag
    specifies the base or default URL for all links on a page and can also specify a common target for all those URLs using the target= attribute. Its purpose is to simplify and shorten the URLs of the links. It also shortens the URLs on the src= attribute used in image, audio, video, and others. There can only be one < base > tag per document (which must be inside the < head > tag).

    Notice in the example below how the linked resources in the < body > section don't have have the first part of a normal URL, because they get that from the < base > tag in the < head > section. They continue on from the last forward slash (/) of the default URL:

    < head >
      < base href=" https://www.w3schools.com/" target="_blank">
    < /head >

    < body >
      < img src="./images/stickman.gif" width="24" height="39" alt="Stickman" >
      < a href="tags/tag_base.asp" > HTML base Tag < /a >
    < /body >
