About CSS

CSS means "Cascading Style Sheets" and it was created by Håkon Wium Lie (co-worker of Tim Berners Lee, the inventor of HTML) and Bert Bos, and was officially released in 1996. Don't worry, "Cascading Style Sheets" also perplexed me when I first heard it. That is why there is a picture of a human focusing mainly on it's skin and muscular system on the left. Like I said on my page about HTML, we can imagine building a web page like the human body.

In order to make a web page first we need a structure on which to hang and hold everything. For a human that part is the skeleton. In the web development world, that structure is HTML. Second, we need to add to that basic structure in order to flesh it out, to bring it to life, to make it look nice. For a human that part is the skin and muscular system, in the web development world that part is CSS.

CSS is another coding language (like HTML) that allows us to modify and style any visible element on our web page within our HTML code. We can control the appearance (aka "styling") of our content including: text size, colour, style, typeface, alignment; link color and style; image size and adjustment; list bullet styles and indentation; table size, shading, borders, alignment and so much more. At first this kind of thing was done in the HTML document, but it made everything messy and complicated, especially when similar styles repeated across multiple HTML documents. Hence why CSS was invented - to allow us to separate design and content and make the web development process easier and simpler.

That is, CSS allows us to make our boring HTML based web page look beautiful and interesting to the user. It is all about the visual appearance of a web page. This is so fundamental to web development that it's virtually impossible to use one of these coding languages without the other. CSS Core Components & Ideas

Fundamental concepts to understand before starting to code with HTML

CSS Syntax: Selectors, Properties, Values

CSS Code typically comprises of three parts:

  1. Selectors (e.g. 'h1', 'p', 'div')

    2. The selector is the HTML tag or element we want to target with our CSS styling rules. The selector is like a bridge: 'bridging' the gap between our CSS code and the HTML code.

  2. Properties (e.g. 'color', 'font-size', 'text-align', 'border')

    3. This is the thing about the HTML tag/element we want to change or modify. There can never be an open space between words; we have to join them with a hyphen like-this!

  3. Values (e.g. 'red', 'large', 'center', 'solid')

    4. This is simply the value for the property we write prior to the value.

The properties and values are separated by a colon (:) symbol, the CSS equivalent of the equals (=) sign. That is, it is like we writing "property=value" but instead using the colon symbol "property:value".


A "declaration" is the name we give to the 'property:value' pair with each declaration separated by a semi-colon (;) symbol. We do this in order to keep them separate and not confuse the browser which is reading the instructions. We can create as many declarations as we like.


All declarations for a selector are contained within curly brackets '{...}', called a "declaration block". This keeps all the code together meaning all these particular styling rules we create will be applied to the HTML tag/element we have chosen, but they will be kept separate from other styling rules and other selectors. So there won't be any conflict of styling rules.

CSS Selectors

As we said in the above section, a CSS selector is the HTML tag or element we want to target with our CSS styling rules. The selector is like a bridge: 'bridging' the gap between our CSS code and the HTML code. However, we can have different types of selectors and it's very important to understand the differences and be able to use the selectors well as they are so fundamental to using CSS:

Quick note: CSS selectors are usually case-sensitive. That is, '.maincontent' is not the same as '.mainContent'. It's important to watch this or else you may see unexpected styling results on your web page!

  1. HTML Element Selectors

    2. Here we simply write the name of element we want to target (or the keyword inside the < tag > if we want to think about it that way) such as 'h1', 'p', 'div' and then we write our declaration block.

    This selector selects all HTML elements of this type/keyword within our HTML document. In general this is not useful because its effect is so wide-reaching when usually we only want to style one or a few elements at a time.

  2. Class Selectors

    3. In our HTML document, inside the opening tag of each element, we can apply a class="" attribute. The value of this attribute is essentially a name or label which we apply to this element and acts as a bridge between the HTML element and the CSS styling intended for it, allowing the styling rules to 'cross' the bridge and take effect. Remember, HTML and CSS are two separate coding languages and so need to be connected somehow in order for the CSS to work with HTML. For example:

    HTML:  < h1 class="header1"> ...< /h1 >

    Then, in our internal or external CSS stylesheet, we put a dot (.) aka the 'class selector' and the value (name/label) we gave our class attribute in the HTML element, followed by curly brackets '{...}'. So this above theory in practice looks like this:

    CSS:  .header1 {...}

    Inside the curly brackets '{...}' go the CSS styling rules which will be immediately applied to the < h1 > element. For example, color: blue; font-size: larger; etc...

    There are some important things to know about the class="" attribute (click the arrow icon to read more about each point):

    It's possible to create multiple classes (multiple class="" attribute values) within the class attribute. That is, one HTML element can belong to several classes and receive several different CSS styling rule-sets.

    For example, a class attribute with the value of "index-h1" or "index_h1" is just one class name (due to the hyphen and underscore lines joining the words together) and would be targeted in CSS like this '.index-h1'.  However, "index h1" are two separate class names and could be targeted separately in CSS like this '.index {...}' and '.h1 {...}'. We might do this in order to separate our styling rules, which we might do because (see next section)...

    It's possible to apply the same class attribute value across multiple elements, and thus apply the same CSS styling rules to those multiple (possibly distinct) elements.

    For example, if we have two < h1 > elements on our HTML document, and we give them both a class attribute with the value "index-h1" (< h1 class="index-h1" >), then both these < h1 > elements will receive the exact same styling rules and will look the same on our web page.

    If we have two < h1 > elements on our document and we want some but not all of the styling rules from the first one to apply to the second one, if we separated the class values ("index h1" not "index-h1") then we can give our second element a class attribute value of "index" and all the '.index' styling rules will be applied, but then create another class attribute value (say "h1_2") with distinct styling rules so our second < h1 > element looks different from the first.

    Furthermore, because we can apply distinct class attribute values to distinct HTML elements, we could also apply our ".index" styling rules intended for the Header 1 elements to a totally different element like a < p > element as well.

    It's possible to target elements inside a parent element with the class="" attribute.

    For example, we have a < div > element (a container element) and inside this container we have a few paragraphs using the < p > tag. The < div > element is the 'parent' element and the < p > elements are the 'child' elements. Instead of adding a class attribute to every single paragraph tag we can just have one class in the < div > tag (e.g. class="indexIntroDiv"). Then, in our CSS stylesheet we write the name of the class as usual, .indexIntroDiv, but then followed by the child element inside the parent element we want to target. So our CSS code will look something like this:

    .indexIntroDiv p {...} ... which is like writing this ... .indexIntroDiv --> p {...}

    And this works because in HTML there is a hierarchical structure. That is, if a parent element is affected or altered in some way, then all its children elements are effected too. For example, in the HTML document the < body > tag is at the top of our visible web page hierarchy. If we add styling to that element then all elements below it will receive that styling too. That is to say, our entire web page!

    Class="" attribute values are case sensitive.

    That is, giving one tag a class="" attribute value of "Index-h1" is not the same as giving another tag a class="" attribute value of "index-h1". If we forget this often this can cause confusion when we don't get an expected styling result on our web page.

  3. id Selectors

    4. The id="" attribute attribute can, like the class="", be applied to any HTML element and works very much in the same way as the class="" attribute. For example, repeating the example from before but using the id="" attribute our example now looks like this:

    HTML:  < h1 id="header1"> ...< /h1 >

    This time, in our internal or external CSS stylesheet, we put a hashtag symbol (#) aka the 'id selector' and the value (name/label) we gave our id attribute in the HTML element, followed by curly brackets '{...}'. So our CSS code will look something like this:

    CSS:  #header1 {...}

    Again inside the curly brackets '{...}' go the CSS styling rules which will be immediately applied to only to this < h1 > element. For example: {color: blue; font-size: larger;} etc...

    There are some important things to know about the id="" attribute (click the arrow icon to read more about each point):

    The id="" attribute's value is unique among all the id="" attributes on the web page and cannot be repeated or reused.

    Unlike the class="" attribute whose value can be applied to multiple HTML tags/elements, the value of the id="" attribute can only be applied to one HTML tag/element. So in our above example, our CSS styling rules will only be applied to the HTML element with the id="header1", we cannot have a second < h1 > element and also apply the id="header1" to it. Instead, we would have to create a different id value such as id="header2". So this means that any CSS styling rules we apply to the an element through the id selector and id attribute will always be unique to that element within our HTML document.

    You can only create one id="" attribute value per HTML tag/element.

    Unlike the class="" attribute where we can create several class values within the attribute, there can only be one value within the id="" attribute.

  4. Custom class/id Selectors

    5. We can also specify that only specific HTML elements should be affected by a particular class. That is, we have a specific element or tag in our HTML document (e.g. < div >) that also has a specific class="" attribute value applied to it (e.g. class="divBig").

    To target these custom HTML elements in CSS we write the HTML element/tag we want to target (e.g. 'div'), followed immediately by the class selector (.) and then the value of the class="" attribute which we have applied to certain 'div' elements (but not all 'div' elements). For example:

    div.big {height: 500px; width: 800px} → all div elements in our HTML document with the class 'big' will appear with these width and height dimensions.

    To make our custom class selector even more specific, we can add another class selector after the first one using the dot (.) symbol, then writing another class attribute value in our HTML document. For example:

    div.big.grayBackground {height: 500px; width: 800px; background-color: gray} → all div elements in our HTML document with the class 'big' and 'greyBackground' will appear with these width and height dimensions and will have a grey background color.

    In addition, we can also add id selectors to our CSS custom class selectors - separated by a space - to make our selector even more specific. That is, we have a specific element or tag in our HTML document (e.g. < div >) that has a specific class="" attribute value applied to it (e.g. class="divBig") and has a specific id="" attribute value applied to it (e.g. id="specialDiv")

    Now, to target this very specific HTML element all we have to do is add the id selector to the end of our previous CSS selector code. For example:

    div.big #SpecialDiv {height: 300px; width: 500px} → now only this element in our HTML document with the class 'big' and id 'specialDiv' will appear with these width and height dimensions (because id attribute values are unique to the HTML element they are applied to).

  5. Universal Selectors

    6. Here we simply write an asterisk symbol (*) in our CSS internal or external stylesheet, followed by our declaration block '{...}' and any styling rules we write will be applied to all the elements in our HTML document. Every single one! The only exception to this is if there is some more specific styling rules applied to an element which overrides the universal selector (e.g. inline-styling). Normally, the universal selector is not that useful and is usually used to remove the default margins and paddings from the elements for quick testing purposes.

    For example:

    * {text-align:center; color: pink} → all elements in our HTML document will be (if the styling rule is applicable to them) centrally aligned and the color pink.

  6. Group Selectors

    7. By writing the comma (,) symbol after our selector in our CSS internal or external stylesheet, we are essentially saying to the computer apply these style rules to X HTML tags/elements or any Y HTML tags/elements. For example:

    Instead of writing in our CSS stylesheet:
    h1 {text-align:center; color: pink}
    h2 {text-align:center; color: pink}
    p {text-align:center; color: pink}

    We can simply write:
    h1, h2, p {text-align:center; color: pink} → It's like we're saying: "Browser, apply these styling rules to all 'h1' elements you find, or any 'h2' elements, or any 'p' elements."

    The same logic applies to class selectors. For example, instead of writing:

    .div1 {background-color:green}
    .green {background-color:green}
    .container {background-color:green}

    We can simply write:
    .div1b, .green, .container {background-color:green}

  7. Child Selectors

    8. We can select only the direct child of an element by writing the greater than (>) symbol and then the child element we want to target. A child element is an element that is directly below and connected to, or contained by, an element in the document tree. In order to understand this logic it's important to understand the theory behind HTML hierarchy and the Document Object Model, which I talked about here.

    For example, in the picture below we have a small HTML DOM tree. We have the standard direct children of the < html > element, the < head > and < body > elements. The direct children of the < body > tag are the < h1 > and < ol > elements, in orange. The < h1 > element has no direct children, but the < ol > tag has four direct children elements, all < li > elements, in blue. One of the < li > has direct children, the < p> and < img> elements, in green. Notice how the change of colour is related to the vertical lines and the connections to the elements at the top of the line.



    Source Code:
    < h1 > Header Element < /h1 >
    < ol >
      < li > 1 < /li >
      < li > 2 < /li >
      < li >
        < p > This is a paragraph element < /p >
        < img src="picture" alt="alternative-text" >
      < /li >
      < li > 4 < /li >
    < /ol >

    So if we wanted to change the styling of all the < li > elements we could write something like this: ol > li {font-weight: 500}

    And if we wanted to change the styling of the < p > element inside the third list item (and any other paragraphs we might create inside list items) we could write something like this: li > p {color: gray}

  8. Descendant Selector

    9. More general than the child selector, the descendant selector targets any descendants of a particular element, whether direct children or not. A descendant refers to any element that is connected to, or contained by, a higher level element in the document tree - no matter how many levels lower it is. For example, all visible elements on our web page are descendants of the < body > element, like the < h1 >, < ol >, < li >, < p > and < img > elements in the image above.

    The descendant selector has no special symbol, you simply put a space between the first and second (or any other) selectors. For example:

    div p {text-align: left; color: white} → all < p > elements that descend from a < div > element receive this CSS styling rule-set. < p > elements that do not descend from a < div > element will not be affected.

    ul.menu li a {text-decoration: none;} → here we only target < a > (anchor) elements that are descendants of list elements which themselves are descendants of < ul > (unordered list) elements that have the class 'menu' applied to them. This avoids us from targeting other < a > elements elsewhere accidentally.

  9. Adjacent Sibling Selector

    10. This selector uses the plus (+) symbol to select the first element that comes directly after another element. Adjacent means "immediately following" and the adjacent element we target must share the same parent element as the first element selector. The syntax is like this: A1 + A2 where 'A2' is the target of the selector. For example:

    p.intro + i {text-decoration: underline} → the first < i > (italic) element after the < p > element with the class attribute value "intro" will have a line under it. The following < i > elements will not, just the first one. A more visual example can be found here.

  10. General Sibling Selector

    11. This selector uses the tilde (~) symbol to select all elements that come directly after another element (or said another way, are proceeded by this other element). It is less strict than the adjacent sibling selector and the syntax is like this: B1 ~ B2 where all 'B2's are the target of the selector. For example:

    p.intro ~ b {color: red} → all < b > (bold) elements after the < p > element with the class attribute value "intro" will be the color red. A more visual example can be found here.

Extra Notes

We have another group of selectors in CSS called the attribute selectors, of which there are several types. Visual examples of all these selectors can be found here. These selectors are useful for styling elements, especially forms, without class or id attributes:

  1. [attribute] selector:
    this selects any elements with this particular attribute. It's like saying to the browser "Style any element that has [X] attribute like {this}." For example:

    [target] {background-color: brown} → all elements that have the target="" attribute applied to them will be styled with this rule-set.

    Or we can be a bit more specific by placing an element selector before the [attribute] selector. For example:

    a [target] {background-color: brown} → now only all < a > elements that have the target="" attribute applied to them will be styled with this rule-set.

    Note: for all the following [attribute] selectors, it's possible to add an element selector before them too.

  2. [attribute="value"] selector:
    this selects elements that have this particular attribute with this particular value. It's like saying to the browser "Style any element that has exactly [X] attribute with [Y] value, like {this}." For example:

    a [target="_blank"] {text-decoration: dotted} → all < a > elements that have the target="_blank" attribute and value applied to them will be styled with this rule-set.

  3. [attribute~="value"] selector:
    this selects elements with a specific attribute which may contain multiple values and we only want to target one of them. In the class="" attribute for example, we can list out many values simply by separating them with a space (e.g. class="div intro leftside") and the same is true for the title="" attribute (e.g. title="flower garden"). It's like saying to the browser "If you find [this] word/value from any HTML element that has [X] attribute applied to it, style it like {this}". For example:

    [title~="flower"] {border: solid black} → all elements that have a title="" attribute which contains the value 'flower' will be styled with this rule-set.

  4. [attribute|="value"] selector:
    this selects elements with a specific attribute whose value must coincide exactly with the value specified in the selector - either the whole world alone (e.g. class="intro") or this word joint to another word but separated by a hyphen (e.g. class="intro-text"). For example:

    [class|="nav"] {padding: 40px} → all elements that have a class="" attribute which contains the exact value 'nav' either by itself or joint to another word separated by a hyphen (e.g. nav-bar) will be styled with this rule-set.

  5. [attribute^="value"] selector:
    this selects elements with a specific attribute whose value starts with a particular value. The value does not have to be a whole word. It's like saying to the browser "Style any element that has [X] attribute whose value starts with the letter/number/symbol [Y], like {this}." For example:

    [class^="1"] {margin: 100px} → all elements that have a class="" attribute whose value starts with the value "1" will be styled with this rule-set.

  6. [attribute$="value"] selector:
    this selects elements with a specific attribute whose value ends with a particular value. The value does not have to be a whole word. It's like saying to the browser "Style any element that has [X] attribute whose value ends with the letter/number/symbol [Y], like {this}." For example:

    [class$="test"] {background-colour: yellow} → all elements that have a class="" attribute whose value ends with the word/value "test" will be styled with this rule-set.

  7. [attribute*="value"] selector:
    this selects elements with a specific attribute whose value contains a specific letter/word/symbol combination. The combination does not have to be a whole word. It's like saying to the browser "Style any element that has [X] attribute which has a value which contains [WYZ] combination, like {this}." For example:

    [class*="ar"] {font-weight: 700} → all elements that have a class="" attribute which has a value which contains within it the letter combination "ar" (for example, class="para") will be styled with this rule-set.

Adding CSS to HTML

When a browser loads a web page, first it reads all the HTML code in order to understand the structure and get the main content for it. Then it reads all the CSS code in order to understand how to style that structure and content and make it appear visually as we intended it to. There are three ways to add CSS code to our HTML document and thus three locations where the browser can look for and find CSS styling for our HTML code:

  1. Inline Styling

    2. So called because we style on the same line as, or 'inline' with, our HTML code. We do this by applying the style="" attribute to the opening HTML tag of an individual element and then writing our CSS declarations ('property: value;') but here without the need for any curly brackets. Instead the declarations go inside the double quotation marks of the style="" attribute, separated from each other by a semi-colon.

    Inline styling only applies to one HTML element at a time. Therefore, it's not very practical for styling whole websites or web pages and, furthermore, it has the disadvantages of not being able to offer all the features that come with our other styling methods (below) like pseudo elements, pseudo classes and media queries and makes our code look messier and harder to read. Because of its high-specificity level it always beats the below styling methods (see "Explaining the CSS name" notes for more information on this logic).

    For example:
    < p style="color:red; text-decoration:underline" > Here will be the individually styled paragraph text < /p >

  2. Internal Stylesheet

    3. So called because we write our CSS styling within, and effect at the same time, the HTML document that we are targeting - 'internally'. Outside this unique HTML document the CSS styling rule-sets have no effect. We create this kind of stylesheet by inserting the < style > tag within the HTML < head > tags at the top of our HTML document. We can use as many < style > tags as we want within the < head > section, as long as they remain within that section. We then connect to the elements in our HTML document using CSS selectors like class, id, child, [attribute] etc.

    This way of styling is of a medium-specificity level because the styling rules can apply to the whole HTML document, but only to this one. Therefore, we should only really use this form of CSS styling if we have a single HTML document with a unique style. If we repeat the same styling rules across several documents, it's better to use the last option in the list.

    For example:
    < head >
     < style >
      .internal-style {
          padding: 15px;
          background: moccasin;
      }
     < /style >
    < /head >

    < p class="internal-style" >
        An element styled with an internal style.
    < /p >

  3. External Stylesheet

    4. So called because we write our CSS styling outside of the HTML document which we are targeting. In fact, we write them in a separate '.css' file, which is loaded into the HTMl document through the < link > tag which goes within the HTML < head > tags at the top of our HTML document. Typically this will look like this:

    < link rel="stylesheet" type="text/css" href="style.css" >

    This way of styling is of a lower-specificity level because the styling rules can apply to the whole website. This is the default way to style our HTML document using CSS because it means we can change many things, located in different places in our HTML documents, from one place and without cluttering our original HTML document. Moreover, changes can easily be made and adjusted through the use of CSS selectors like class, id, child, [attribute] etc.

Explaining the CSS Name (Specificity)

CSS stands for "Cascading Stylesheets." But what does that mean?

It might be helpful to think of a 'cascade' (or more commonly, a waterfall) when thinking about the meaning of CSS's name, hence the above picture. The question we have to ask ourselves when we start to learn about CSS is:

When a page includes multiple external, internal, and inline style rule-sets, which rule-set takes precedence?

The way CSS works is very much like a 'cascade' or waterfall in that it reads rule-sets and instructions from top to bottom. For example, if we have two < link > tags in our < head > section which link to external CSS stylesheets, CSS would use the second one that appears because it is the last rule-set it found before displaying the HTML and so it overrides previous CSS code. The second stylesheet is closer to the HTML it's styling, it's more specific. Another example is if we have to individual rule-sets for an HTML element (e.g. a < h1 > element) within an internal stylesheet. Imagine we write 'h1 {color:green]' but then further down that stylesheet we write 'h1 {color: yellow}'. CSS would use the second rule-set because reading from top to bottom that it the more specific and closer rule-set than the 'h1 {color: green}' one.

Typically the CSS 'cascading' works like this:

First, the browser starts with the most distant/less specific stylesheets for the HTML code. The most distant stylesheet in this case is the browser stylesheet. In fact, all browsers have their own default styles (a list of which can be seen here). So then, without even inputting any CSS code ourselves, our web page will already have default color settings (e.g. a white background with black text), font style, will automatically configure headings and text sizes etc. This is so the browser can initially display a web page and its content to the web developer.

⬇

The next most distant/specific stylesheet is the external CSS stylesheet (a separate '.css' file). These are custom styling rules and for that reason are more specific but, moreover, they are more closely located to the HTML code and for these two reasons the external stylesheet overrides the previous browser's default stylesheet.

⬇

Then, we have the internal CSS stylesheet by using the < style > tag within the < head > section of the same HTML document itself. Again, these styling rules are more closely located to our HTML code and are again more specific (internal stylesheet rule-sets only apply to the HTML document in question whereas external stylesheets can apply to multiple documents) and thus override the external stylesheet rule-sets.

⬇

Finally, we have the inline styling by using the style="" attribute. These styling rules are the closest to the HTML elements being targeted (right next to them in fact) and very specifically tailored for them too. So these styling rule-sets, if present, override all the other styling rule-sets we see above.

So then, CSS is a 'cascading' system of styling rules, evaluating rule-sets for their specificity and distance from the HTML element being styled. The closest and most specific CSS rule-set for the element it is trying to style 'wins'.

For example, imagine we have a paragraph < p > element in our HTML document. And we have these styling rules for it:

Default Browser Stylesheet: {color:black}
External Stylesheet: p {color: green}
Internal Stylesheet: p {color: red}
Inline style: < p style="color: blue" >

In this case the inline style is the closest rule-set to the paragraph element it's targeting and 'wins' over, or overrides, the other two styling rule-sets and the text appears blue. If we took away the inline style the text would be red. If none of these custom styling rule-sets were present then the browser would default to its own stylesheet, the furthest away and least specific styling rule-set, making the text black.


However, it can get a little more complicated if we include CSS selectors because these too have an effect of CSS levels of specificity! Here is a list with the general order of specificity for CSS selectors:

  1. Inline styles
  2. Id selector
  3. Class, pseudo-class, and attribute selectors
  4. Element and pseudo-element selectors

So that is, if we have external stylesheet which has a rule-set "h2 {color: pink}" and within the same stylesheet a rule-set which says ".h2Black {color:black}" then this second rule-set will be applied to this one < h2 > element because the class selector makes it a more specific rule-set than the generic element. A helpful article has been written about specificity and understanding which CSS selector will override other selectors here. CSS Units

It is essential to know about and understand CSS Units because they are involved with, or are fundamental to, pretty much everything we do with CSS.

CSS units allow us to style and size things exactly as we want, often expressed as "length" values; that is, a number followed by a length unit e.g. 10px, 2em, etc. They represent a distance or size value There is no gap between the number and length unit (except if the number is 0, then there is no length unit).

In CSS we have two kinds of length units, within which we have multiple length unit options. However, only a few length unit options are common and widely used on the web:


ABSOLUTE LENGTHS

These length units are fixed, meaning we can set a precise length which will always display as that size regardless of the device or screen it is being displayed on. That is, a < div > with a length of '1cm' will measure exactly 1cm wide, always, whether on a big screen or a small one.

Pro: good for physical media like papers, posters and other printed work as we can reliably know how big or small something will appear when printed.

Con: as they don't adapt in size to device or screen differences and always stay the same, they're not so good for making our web pages responsive - something which is key in this day and age. That is, if a user wants or need to change the font size of the browser or computer (for example, they having sight problems), these length units don't scale, so the user has to zoom the whole screen in and cut off half the page.

Pixels (px)

  • Our screens are made up of pixels or 'dots'. 1px = 1 dot. The pixels our screen has, the higher its resolution.

  • The total amount of pixels on a screen varies from device to device. 1px on a big screen seems a lot smaller than 1px on a small screen. A 200px wide < div > container will appear big on a small screen but small on a big screen. So really, 'pixels' as a unit of length is relative.

  • Is by far the most common absolute length unit and still widely used by web developers.

Inches (in), Centimeters (cm), Millimeters (mm)
Although these absolute length units exist, they are rarely used in the web dev world and have the same advantages and disadvantages as any other absolute length unit.

RELATIVE LENGTHS

These length units define a length which is relative to another length property. We need to be aware of which length property they are relative too, otherwise, we will get unexpected results.

Pro: are good at adapting and changing with different device or screen sizes. Allow users to change the font-size of the browser if they have sight or reading difficulties.

Con: not so good for printed media as we can't be sure exactly how the elements will appear on the printed page, and require a little more brain power and occasionally maths in order to get the correctly proportioned appearance.

Em

  • The length is relative to the parent element. That is, if the length property of a < div > container is 10px, then for the < p > element within that container 1em=10px, 2em=2px etc.

  • Can get complicated when used for objects within objects within objects etc because the font-size, always based off the object previous (i.e., the parent element), can potentially compound in size with each new layer, producing bigger and bigger font.

  • Is good to use for an object so that it has padding or margin that scales with the size of the element (e.g. buttons). That is, the paddings and margins will always be in proportion.


Percentage(%):

Works in the exact same way as 'em', just represented in percentages. For example, 50% = 50% of the parent element, or 0.5em.

Combining em and %: The < body > tag is the parent element for all our visible web page. If we set a length property (in px), then on our page 1em=body element length (e.g. 20px). So, handy trick used by developers to simplify things is to set the font-size for the body element to 62.5% (62.5% of 16px), which equates to 10px, or 0.625em. Now you can set the font-size for any elements using em units, with an easy-to-remember conversion, by dividing the px value by 10. This way 10px=1em, 12px=1.2em, 14px=1.4em, 16px=1.6em, and so on.

Root em (rem):

  • The length is relative to the < html > or 'root' element at the top of our DOM tree model. That is, it doesn't matter where our element is within the HTML DOM tree, the length of the element we're editing will always be compared to the < html > element at the top of the document. By default, the normal browser font-size is 16px, so 1rem = 16px.

  • Is a very dependable length unit because the root element length is always the same. We know what we're getting when we write '1rem' or '2rem'.


Viewport Width (vw) and Viewport Height (vh):

  • The length is relative to the entire width or height of the browser window size. That is, it doesn't matter where our element is within the HTML DOM tree, the length of the element we're editing will always be compared to the browser window size. 1vw=1% of the viewport's width and 1vh=1% of the viewport's height.

  • Is a good length unit for responsive web page design, because the size is always relative to the device or screen size. That is, if the browser size increases, the length of an element or of our font increases, if the browser size decreases, they decrease.

Color Gradients

The 'background-image:' property has another interesting use which allows us to make more visually interesting and dynamic backgrounds for our elements and web pages. With this property we can make color gradients, meaning, color blends. There are three types:

Linear-gradients

We make linear or 'line' gradients by writing the 'background-image:' property and then giving it a value of 'linear-gradient (X, Y)'. In the brackets we specify color-stops, meaning, the colors we want to render smooth transitions among - separated by a comma. We can use any of the color selection methods mentioned before in the colors section to give us our color values. For example:

This is a div container with a linear gradient background 'image'. CSS code:

{background-image: linear-gradient(red, yellow);}

By default the colors transition smoothly from top to bottom, represented by the value 'to bottom'. However, we can control the direction of the transition by using keyword values or degrees. We put these keywords or degree values before the color-stop values, within the brackets, also separated by a comma. That is: 'linear-gradient (keyword/degrees, X, Y)'. Below is a list of these keywords and their degree equivalents (use the degree circle diagrams to help you understand what's happening in practice) although if we want to we can write out custom degree values too:

'to top' = '0deg' → the first color (red) starts at bottom and moves towards top.

'to top right' = '45deg' → the first color starts on the bottom left and moves towards the top right.

'to right' = '90deg' → the first color starts on the left and moves towards the right.

'to bottom right' = '135deg' → the first color starts on the top left and moves towards the bottom right.

'to bottom' = '180deg' → the default setting (from top to bottom).

'to bottom left' = '225deg' → the first color starts on the top right and moves towards the bottom left.

'to left' = '270deg' → the first color starts on the right and moves towards the left.

'to top left' = '315deg' → the first color starts on the bottom right and moves towards the top left.

We can also control when the colors transition into each other. By default as soon as the color appears it starts transitioning into the next color. Specifically, if we don't specify the location of a color stop, it is placed halfway between the previous color and the next one that follows it. However, to change this we can apply a percentage value after our color value which tells CSS to apply color A up until X point, and only after this point does it start transitioning into color B. For example:

A div container with two colors at 50%. CSS code:

Fonts

A font is a set of printable or displayable text characters in a specific style. Choosing the right font and style is very crucial for the readability of text on a page, it can make a big impact on how readers experience a website, and it can create a strong identity for a brand. In short, the font we pick for our website is very important!

First, before we look at the different CSS properties for changing a text's font, let's try to understand more about fonts in general:

There are five generic font families, which short explanations about font psychology to go with them, examples of which can be see here.

  • Serif: fonts have a small stroke at the edges of each letter. They create a sense of formality, elegance, heritage, and authority.

  • Sans-Serif: fonts have clean lines (no small strokes attached). They create a modern, minimalistic, straightforward and sophisdicated look.

  • Monospace: here all the letters have the same fixed width. They create a mechanical look and is most commonly used to display code.

  • Cursive: imitate human handwriting. They feel more personal but rely on context to get their point across. They can create a formal, elegant feel or equally a more fun, playful feel - depending on the cursive font.

  • Fantasy: decorative/playful fonts, but not widely available across different browsers and operating systems.

  • A more detailed look into font psychology can be found here.

The 'font-size:' property sets the size of our text on the web page. Understanding text size is important in web design. However, we should make sure to use the proper tags and elements instead of abusing CSS to get the appearance we want. That is, we should not use font size adjustments to make paragraphs look like headings, or headings look like paragraphs etc but instead use the < h1 > tag or < p >.

By default, the normal browser font-size is 16px (16px=1em). However, in order to alter this we have three different methods of sizing our text, each with different advantages and disadvantages.

ABSOLUTE FONT-SIZE

This allows us to precisely set a font-size that will not change regardless of the device or browser being used. The user cannot change the font-size within their browser - which is bad for accessibility reasons. Is useful when the physical size of the output is known (e.g. printing). Below is a selection of the different measurement values we can use to produce absolute font-sizes:

  • Pixels (px): this is a good choice when you need the pixel accuracy. However, it's not very accessible because users with limited or low vision may wish to enlarge the font-size and they won't be able to. It is possible to user the 'zoom' feature, but this resizes the entire page and so the user can't see the rest of the page when they zoom in. Therefore, it's best to avoid pixel values.

  • Absolute Keywords: here we just use certain keywords in order to size the text whose value is relative to the < html > root element. We saw before that the default browser text size is 16px, which is the equivalent here to 'medium'. These words are: 'xx-small' (9px), 'x-small' (10px), 'small' (13px), 'medium' (16px), 'large'(18px), 'x-large'(24px), 'xx-large'(32px). Using this method, by setting a font size on the < body > element, we can set the relative font sizing everywhere else on the page, giving us the ability to easily scale the font size up or down accordingly.

RESPONSIVE FONT-SIZE

This allows us to set a font-size that is relative to the size of the browser window. That is, if the browser size increases, the font-size increases, if the browser size decreases, the font-size decreases. We have two measurement values we can use to produce responsive font-size:

  • Viewport width (vw): here 'viewport' means the browser window. 1vw=1% of the viewport's width.

  • Viewport height (vh): again 'viewport' means the browser window. 1vh=1% of the viewport's height.

However, the problem with the responsive font-size method is that on small screens fonts become so small that they are hardly readable! To prevent this we can utilize CSS calc() function (see here) or we can use media queries.

RELATIVE FONT-SIZE

This allows us to set the font-size relative to the surrounding elements and is good for accessibility reasons because this method does allow a user to change the font-size within their browser. Below is a selection of the different measurement values we can use to produce absolute font-sizes:

  • Em: this unit of measurement refers to the font-size of the parent element where 1em is the same as the font of size of the parent element. Using the default browser font-size, 1em=16px, 2em=32px etc. To get our em value we simply have to divide pixels by 16 (px/16=em).

    Note: The < body > tag is the parent element for all our visible web page. If we set a font-size here (in px), then on our page 1em=body element font-size (e.g. 20px). Also, within each new child element if we put the font-size as 1.2em the text will gradually balloon in size as each element's font-size grows on the last. Lastly. we can use 'em' for the sizes of margins and paddings which meaning their size will be relative to the element's font-size. This means regardless of whether the user is on a big or small screen, the sizes of the paddings/margins etc will be in proportion

  • Percentages: percentages act in the same way as 'em' values because the font-size also is relative to the parent's font-size. So 110%=1.1em, 90%=0.9em etc.

  • Combining percentages and em: a handy trick used by developers to simplify things is to set the font-size for the body element to 62.5% (62.5% of 16px), which equates to 10px, or 0.625em. Now you can set the font-size for any elements using em units, with an easy-to-remember conversion, by dividing the px value by 10. This way 10px=1em, 12px=1.2em, 14px=1.4em, 16px=1.6em, and so on.

  • Rem: or 'Root em' is different from 'em' because the element's font-size is always relative to the < html > root element regardless of where the element is within the DOM tree model. That is, the value remains context independent. Typically, the root element font-size is 16px so 1rem = 16px.

  • Relative Keywords: here we just use the keywords 'smaller' or 'larger' to specify a larger or smaller size relative to the parent element. If the parent element's font size is 20px, then setting the child's font-size to 'bigger' will mean its font-size is slightly more than 20px, and setting it to 'smaller' will mean its font-size is slightly less than 20px.


The 'font:' property is a shorthand version that allows us to set all the individual font properties in one property. In this shorthand version, the 'font-size' and 'font-family' values are required. If one of the other values is missing, their default value is used. The order of the values for this property go as follows (without the dashes): font-style - font-variant - font-weight - font-size/line-height - font-family. For example:

This is an example of a sentence styled using the 'font:' shorthand property.

The Box Model

The CSS term 'Box Model' is used when talking about design and layout. According to CSS, all HTML elements are essentially boxes with multiple layers wrapping around it. This box consists of the content area (the content of the HTML element), and then optional margins, borders, and padding areas.

As this box model applies to all HTML elements, and CSS affects all HTML elements, it's fundamental to understand this model. It helps us structure and layout our web page, as well as understand the relationship between our different HTML elements. When used well, the box model can give our web page structure and make it more visually appealing to the user due to the use of spacing.

The following diagram demonstrates how the width, height, padding, border, and margin CSS properties determines how much space an element can take on a web page.

Here's an explanation of the different parts:

  • Content: where content of the HTML element, i.e., where text, images and other content appear - stylable.

  • Padding: a transparent layer that clears an area around the content - not stylable.

    Note: if the content area has a background-color then it will be visible through the padding area also.


  • Border: a border which goes around the padding and content sections - stylable.

  • Margin: a transparent layer that clears an area around the border section - not stylable.

    Note 1: if the parent element has a background-color then it will be visible through the margin area of the child element.

    Note 2: the margin only protects its own element. That is, if a margin finds another element's margin, the two margin's simply overlap, they don't press up against each other, they don't add their combined lengths. So then, if we have two neighbouring < div > containers and one has a margin of 50px and the other has a margin of 30px, the margin between the two is not 80px! The margin between them will be 50px because the bigger margin is defining the distance between the two containers, it's protecting its own element.


The Height and Width of a CSS Box

Something that is a little confusing is how to calculate the total area (width x height) of the box. Imagine we have a < div > container and we make it measure 100px width by 100px height using the 'height:' and 'width:' properties*, for example:

However, the default value is 'auto' meaning the width and height automatically adjust to allow the content to be displayed correctly. In general, we should avoid setting a fixed width and height unless it is necessary.

Currently the box has a total area of 100x100px which we can see in the image below. This makes sense because we set our height and width to be this.



However, watch what happens when we start adding additional padding (20px), border (5px) and margin (10px) lengths:




We can see that now the box's total area (width x height) is 150px! That is because the 'height:' and 'width:' properties only apply to the content area of our CSS box. Additional box sections get added on top of this.

So if we want to calculate the total width of a box, we have to perform this calculation: width + padding-left + padding-right + border-left + border-right + margin-left + margin-right = total width.

And if we want to calculate the total height of a box, we have to perform this calculation: height + padding-top + padding-bottom + border-top + border-bottom + margin-top + margin-bottom = total height.

But we only wanted a box of 100x100px! This means we would have to tediously cut down on each section and do constant calculations until we get a box which perfectly fits the 100x100px proportions. That would be time-consuming!

So in order to get round this we can use the 'box-sizing:' property which takes two values:

  • content-box: the 'width:' and 'height:' properties only apply to the inner content box of the element's box model. Additional padding, border and margin lengths get added on top of that in order to give us the total box height or width.

  • border-box: the 'width:' and 'height:' properties apply to all parts of the element: the content, the padding and the borders. Whatever width or height we set with the 'width:' and 'height:' properties will be the full width/height of the element with each box section adjusting to accommodate to these limits.

Putting this property into action, now notice what happens to the div container with the 'box-sizing: border-box' rule-set declaration applied - the box now has a total area of 100x100px and to accommodate this the content area has been reduced in size, allowing the padding, border and margin sections to squeeze in.

Tip: to make our lives easier, when starting a project, we can use the universal selector (*) in our CSS stylesheet and apply the rule-set: 'box-sizing: border-box' so that automatically our boxes work this way. This way we should never get confused about what is happening on our web page such as when we set a box to '100%' width, then add padding or margin and find the box is wider than our web page!



Two other properties that we might want to use when editing the total size or area of our box are the 'max-height:' and 'max-width:' properties.

These properties define the widest and highest possible length values for an element. If the content is larger than the max-width, the content will automatically change the height of the element to adjust. If the content is larger than the max-height, it will overflow (requiring the 'overflow:' property to control this). If the content is smaller than the max-width or height, then these properties have no effect. Note: in both cases the 'max-' property overrides the normal property, meaning, if the width is 300px and the max-width is set to 200px, then the element's actual width will be 200px.

In a similar way, we can apply to our elements the 'min-height:' and 'min-width:' properties which establish a min-height or width size for the content box. This value takes precedence over the normal values. That is, if the width is 300px and the min-width is 400px, then the element will be 400px wide not 300px.

Often, these min and max height and width values are used together in order to create a width and height range, aiding flexible design.

The 'position:' property

Positioning elements appropriately on the web pages is a necessity for a good layout design. The 'position:' property allows us to specify the position of an element on our website but without having to use margins and padding. We can choose from a selection of 5 different values: static, relative, fixed, absolute or sticky.

Positioned elements don't participate in the standard HTML document flow but instead specify their position relative to a container element. For example, usually if we edit the padding part of a container, then this change will have an impact on other elements around this element, because it's part of the normal document flow. However, these positioned elements are removed from that flow, allowing them to be moved independently and not have an impact on other elements on our page.

A positioned element uses the 'top:', 'right:', 'bottom:', and 'left:' properties to align itself inside its container, but only once the 'position:' property has been applied. We apply lengths to these properties to position the container, which is typically done in pixels but other lengths can be used too.

Below is an explanation of each value for this property:

  1. 'static' (default): elements positioned this way are not affected by the top, bottom, left, and right properties. They're not positioned in any special way, but are simply positioned according to the normal flow of the page. For example:

    A div with the 'static' position.

  2. 'relative': these elements are positioned relative to its normal position, according to the normal flow of the page, and doesn't effect other HTML elements. So at first the element will be positioned as normal, but by specificizing a length with the following properties: 'top', 'bottom', 'left', 'right', 'z-index' we can move it around relative to that position. For example, the style "left: -30px;" subtracts 30 pixels from the element's original left position, meaning it will move across to the left; the style "left: 50px;" adds 50 pixels to the element's original left position, meaning it will across to the right. Elements with 'position: relative' are often used as containers for 'position: absolute' elements, acting as an anchor point for them.

    Note: a relatively positioned element can be moved and overlap other elements, but it keeps the space originally reserved for it in the normal flow.

    A div with the 'relative' position and 'left: -30px;', 'top: 20px;'.

CSS Flexbox

Before flexbox web developers relied heavily on the 'float:' and 'position:' properties in order to determine the layout of their webpages. However, these weren't so much designed for this function. Therefore, the flexible box or 'flexbox' was born!

Normally with other 'display:' property values we define element layout and positioning in relation to other elements or the normal web page flow. However, with the 'display:flex' value we create a block-level element, a container, whose main purpose is to control how the children inside the container behave. Note, this doesn't include nested children elements which remain unaffected within the flex container.

Furthermore, normally within our document we can have a multiple dimension layout, setting one element to have 'display: block' and the next to have 'display: inline'. However, flexbox is a one-dimensional content layout model, meaning the content is only displayed in one way; or flexbox children (referred to as 'items') are displayed in a row-type fashion (from left to right, next to each other) or in a column-type fashion (from top to bottom, stacked on top of each other). But that's it!

These items then may automatically expand ('flex') and shrink to fit the available web page space, or adapt to different screen sizes, or prevent overflow etc. Flexbox is especially adept at handling items of different sizes and returning the best layout for those items. Instead of setting rigid dimensions for the browser to follow, with flexbox, we can instead provide flexible boundaries to hint how the content could display. This flexibility means flexbox is a great tool for creating responsive web pages.

In general flexbox is more appropriate for smaller-scale layouts, for organising sections of a website, whereas CSS Grid (see in the CSS Grid section) is intended for larger scale layouts or indeed whole web pages.

Some important things to note or bear in mind with flexbox:

  • The 'float:', 'clear:', and 'vertical-align:' properties have no effect on flex items.

  • Margins don't collapse/overlap inside the flexbox, they bump up against each other. That is, it's not the largest margin that determines the space between the elements like what normally happens. It's the combined margin lengths which determine the distance between them.

  • Flexbox items always work along the main-axis (the direction in which the elements are going in) and cross-axis (the axis perpendicular to the main-axis). Below is an image to help us understand how this flexbox logic works:



    When the items are set out in a row-type display, the main-axis goes from left to right (horizontal). When the items are set out in a column-type display, all we need to do is rotate the image 90 degrees so the main-axis is pointing downwards because the main-axis now goes from top to bottom, is vertical.

    Note: it's possible to reverse the main-axis direction from 'left to right' and 'top to bottom' to 'right to left' and 'bottom to top' - see 'flex-direction:' property notes.

  • It's possible to change the flexbox container from a block-level element to an inline-block element. All that's required for this is to apply the 'display: inline-flex' property meaning the container behaves like an inline block within the document, and the children behave like flex items within the container.

CSS Grid

CSS Grid is similar to CSS Flexbox, but is basically the next-level version of it.

It's similar because we are creating a block-level container via the 'display:' property (this time with a value of 'grid') whose main purpose is to control how its children inside it behave. Again, this doesn't include nested children elements which remained unaffected within the grid container, and again this block-level container can be changed to an inline-block level container by applying the 'display:inline-grid' value.

The difference between the two is that grid is a two-dimensional content layout model, meaning the content can be displayed in two directions at the same time. Items can be displayed as rows and as columns simultaneously within the grid container.

CSS Grid offers page layouts that are supported by all modern browsers, and excels at dividing a page into different major areas and specifying the size, position or layer of these areas. It's similar to an HTML table but allows us to do a lot more than this traditional layout design (e.g. items can overlap each other within a grid container).

Considering CSS Flexbox and CSS Grid are similar, which should we use when, and why?

Flexbox

  • Smaller-scale layouts, necessarily flexible layouts.

  • One-dimensional layouts (either rows or columns).

  • Layouts of content with different sizes as it gives bigger items more space and smaller items less space, preserving readability of content. It makes different sized content look good.

  • A container with a layout that in the future would allow for future content to be inserted into it.

Grid

  • Bigger-scale layouts, perhaps even whole web pages.

  • Two-dimensional layouts, table-like layouts (with rows and columns). Therefore, the layout doesn't have to be just a big layout in order to use grid. Any two dimensional layout of any size is better with grid than flexbox.

Media Queries

*Have later discovered that @media is one of CSS's 'conditional rules'. Here is used to apply styling based on the media queries, IF they're met.

A media query is a piece of code that constantly 'queries' or evaluates data about the device or devices accessing the web page currently. That is, the "media" form currently accessing the web page (hence the name 'media' query).

A media query can evaluate all of the following things:

  • device type: 'screen', 'print', 'speech', or 'all'

  • width & height of the viewport (browser window)

  • width and height of the device (screen size of device itself)

  • orientation: landscape or portrait

  • screen resolution

  • color depth

Media queries are important for us web developers because they improve user experience and are essential for responsive web design. Media queries allow us to customise our web page presentation based on the specific device accessing the web page, or certain conditions on that device. For example, if the device is a mobile phone, a tablet, a desktop, the browser program viewing the website, the system settings of the device etc.

Media queries work on a true or false basis. That is, if the condition of the media query is met then its styling code block is applied to our webpage. If the condition is false, then no code block is applied.


Media Query Syntax

  • at-rule: defines the media query and essentially acts as a logical operator, meaning 'if'. That is, 'if' X condition is met, do Y.


  • media type: the type of media we want to target. There are several values we can apply here (although 'screen' is the typical one we use):

    • all (default): Matches all devices. When no media type is specified 'all' is the default or presumed value.

    • print: Matches documents viewed in print preview mode or any media that breaks the content up into pages intended to print.

    • screen: Matches devices with a screen.

    • speech: Matches devices that read the content audibly, such as a screenreader.

      • Note: multiple media types can be used simultaneously, separated by commas.


  • media feature: now we know the media type we're targeting, we can start defining what features we are trying to match it to. Media features are defined within parentheses. The typical example is to match screens to certain widths where 'screen' is the media type and both min-width and max-width are features with specific values.

    A detailed list of the different media features we can define can be found here.

    Note: 'min-' values mean including and above X value and the 'max-' values mean up to and including X value. They are constraint conditions. So "@media screen (min-width: 300px) and (max-width: 1000px) {...}" means that X styling will apply to screens/browser windows from 300px up to and including 1000px, but not outside this range. Those are the conditions of this media query.

  • operator: allows us to combine media types with media features and in different ways, making the conditions required for the media query to apply more complicated. There are several values we can apply here:

    and
    1) allows us to combine media types and media features together;
    2) allows us to specify multiple media feature requirements by placing this operator between media features (e.g. screen width ranges).

    E.g: @media screen and (min-width: 481px) and (max-width: 960px) {...}

    'or' (commas)
    the media query becomes a list of queries which will apply if any of the comma separated queries matches.

    E.g: @media (orientation: portrait), (min-width: 3rem) and (max-aspect-ratio: 2/1) {...} = media query will be true if the device has an orientation of portrait OR if the device's viewport has a min-width of 3rem and a max-aspect-ratio of 2/1.

    not
    allows us to apply certain styling for certain devices which do not support or match certain conditions. Note:
    1) with this operator defining a 'media type' is obligatory
    2) 'not' only negates one query, not an entire query list (queries separated with commas).

    • E.g: @media not screen and (hover: hover) {...} = code block will apply when the primary pointing device can't hover on elements.


  • Here are some more examples of simple media queries:

    @media print {...} = styling for print media only

    @media (max-width: 65rem) {...} = styling for all devices that have a display width of 65rem or less

    @media screen and (min-width: 800px) {...} = styling for screen viewports that have a width of 800px or more


  • CSS Code: although it's not included in the image, obviously we need to add our CSS code block after the @media query is finished! If we want to target a specific element which has a class or id selector, inside the first set of curly brackets we write the name of this selector, followed by another set of curly brackets, then the different CSS code we want to apply if the given conditions are met.

CSS Animations: Transitions

CSS transitions allow you to smoothly 'transition' between two different values for one or multiple properties, animating the changing values over X period of time.

In technical terms this process of creating "in-between" steps that smoothly transition from one state to another is called "interpolation".

In order for a transition to work we must specify two things:

1) a CSS property to target/change (e.g. color, border-radius, width etc)

2) a duration for the transition effect either in seconds or milliseconds**

*1: If no duration is set, nothing happens becaus the default value is 0.
*2: It's best to use milliseconds because Javascript only accepts this value.


Note:

  1. Apart from the above two requirements for transitions, all other transition properties are optional.

  2. Not all CSS properties are animatable but, in general, any CSS property that accepts values that are numbers, lengths, percentages, or colors is animatable.

Here is a list of our different CSS transition properties and a description of what each one does:

  • 'transition-property:' (obligatory) - specifies the name or names of the CSS property or properties which the transition effect should be applied to. For multiple properties, each property must be separated by a comma.

    Note: Not all properties are animatable (as I've learnt the hard way!). Check this list to check the property or value you're targeting is animitable if your transition animation isn't working.

  • 'transition-duration:' (obligatory) - specifies the number of seconds or milliseconds a transition animation should take to complete. Again, milliseconds is preferred here.

  • 'transition-timing-function:' - specifies how the intermediate values of the CSS properties being affected by a transition will be calculated. Essentially, this property effects the visual apperance of the transition: fast? slow? slow then fast? fast then slow? etc.

  • 'transition-delay:' - specifies when the transition will start ... after how long a time?

  • 'transition:' - the shorthand property which specifies the above four properties together, in the above order. For example, the following in the shorthand form is the same as writing in the longhand form:

    “transition: width 2s linear 1s;”
    =
    "transition-property: width;
    transition-duration: 2s;
    transition-timing-function: linear;
    transition-delay: 1s;"

    Note: if we don't specify a value in the shorthand property, the default value will be used. The default values for the shorthand property are as follows: "all 0s ease 0s". That is, if we don't set a time value here nothing will happen, because the default value is 0!

So how do we actually apply transition animations to our elements? There are two methods:

METHOD #1

Part 1:
We write some CSS styling(s) for an HTML element via the element/class/id selector etc. Within the CSS declaration block we include the 'transition:' or 'transition-duration:' property and we only apply a time value to our property.

Part 2:
We target this same HTML element again, but add a pseudo-class to it such as ':hover', ':focus', or ':active' etc. That is, when this element is in another state. In this new block we include the same properties as before (or at least the ones we want to change in the new element state) and give them the different values they will have after transitioning. By default, without any specific property or properties specified, the browser will try to transition all properties according to the time-value we gave when the pseudo-class state is activated.

Here is a live example, with the code written below.

Notice how all the properties with different values automatically change when you hover the mouse here.

                             color: black;
                             border: 2px solid black;
                             transition-duration: 1000ms}

                            .container:hover
                            {background-color: black;
                             color: white;
                             border:2px solid yellow;}

METHOD #2 (Preferred*)

*preferred because the less properties we animate simultaneously the better, because it eats up CPU and potentially GPU power, slowing down the website for the user.

Part 1:
Like before we write some CSS styling(s) for an HTML element via the element/class/id selector etc. Then, within the CSS declaration block we can either:

1) include the 'transition:' shorthand property and declare both the property/ properties we want to change AND the time value within the same declaration;

2) write out and give values for both the 'transition-property:' AND 'transition-duration:' properties.

Part 2:
The same as part 2 for method #1.

  • linear: the animation has the same speed from start to end.

CSS Animations: Animations

CSS Transitions and Animations have a lot in common, and in a lot of ways are similar. However, there are some key differences between CSS transitions and animations:

  1. Transitions need to be triggered, animations do not.

    That is, we need to apply a state change to an element in order for a transition to take place - transitions are reactive - whereas an animation can start up as soon as we open the website.

  2. Transitions are designed to run just one time, animations can loop infinitely.

    When triggered, a transition occurs and then finishes. In contrast, an animation can occur over and over again as many times as we desire.

  3. Transitions are designed to go from a start point to a finish point over X period of time, whereas animations give us more precise control over the changing of property values, which property values change, and when they occur.

    With transitions we can only really control the time it takes for the property values to change from A to B and how they change via applying different values on the bezier curve. In contrast, with animations we have keyframes which allow us to precisely control what properties change and when they change. That is, we can establish intermediary points or 'animation steps' in the animation and change what happens at each point/step.

  4. Transitions are more Javascript friendly than animations.

    Because transitions are well-defined they are more easy to modify with Javascript code, however, animation keyframes are more difficult and complicated to modify.

Below we have a visual example of the difference(s) between transitions and animations:

So in general, transitions are better for simple, reactive animations and for when we want to use Javascript to manipulate them, and animations are better when we want more precise control over the animation and what happens within it (or indeed before and after it), and for infinite looping animations.

We also have to take into consideration that animations that effect width, height and other sizing properties will have an impact on the whole of the webpage and its elements and thus the CSS values for all the other elements. This could cause layout problems or use up a lot of computing power, so it's best to avoid using transitions for sizing property changes, (see the 'Transform' notes below). On the other hand, transitions are usually more website and computing power friendly.

So now we understand the difference between CSS transitions and animations, let's look at the essential and differentiating piece of CSS animations: keyframes!

What are keyframes?
They are timestamps, markers on the timeline of a piece of media, that contain information or instructions for what should happen or change to the media in question at that moment on the timeline in which we put them. In this case, they contain instructions for what styles an HTML element will have at certain times.

We establish keyframes independently from elements and selectors and they only take effect when we link them to said elements and/or selectors.

Within a keyframe we can write as many 'animation steps' as we like and change any number of properties that we like as well.


How do we create them?
This is the keyframe syntax:


We have to give our @keyframe a name. This is the keyword we use to link it to our element or selector later.

Then we write 'from' followed by a code block specifying the starting properties and their values, and then 'to' followed by a code block which specifies the ending properties and their values.

If we want to, we can write percentage based timestamps inbetween these values to have more control over the animation. This, or, we can use just percentages for all our timestaps. 'From' is the same as writing '0%' and 'to' as writing '100%'.

How do we link our keyframes to our elements and selectors?
1) By using the 'animation:' shorthand property and writing as the first value the name of our keyframe (e.g. @keyframe my-first-animation {}"), followed by a duration value such as "500ms".

2) Applying separately the 'animation-name:' property followed by this same keyframe name, and then the 'animation-duration:' property followed by a time value.

Here is a list of our different CSS animations properties and a description of what each one does. These properties are applied directly to the element or CSS selector code block:

  • 'animation-name:' (obligatory) - here the value is the name of the @keyframe we independently established elsewhere in our CSS. This property is obligatory because without it the keyframe code block won't be connected to this element/selector and it won't have any effect upon it.

  • 'animation-duration:' (obligatory) - like 'transition-duration', this specifies the number of seconds or milliseconds a transition animation should take to complete, the main difference being here that negative values are accepted. Remember, milliseconds is preferred here for Javascript reasons.

  • 'animation-timing-function:' - like 'transition-timing-function', this specifies how the visual apperance of the transition and uses values like 'ease', 'linear', 'step-end' etc. A complete list of the values available for this property can be viewed here.

  • 'animation-delay:' - like 'transition-delay', this specifies when the animation will start ... after how long a time?

  • 'animation-iteration-count:' - says how many times we want the animation to run. We can either give numeric value (e.g. 1,2,3 etc) or the keyword 'infinite' meaning the animation will loop continuously forever.

    Note: once a numerical value iteration-count has finished, we have to refresh the page in order to experience the animation again.

  • 'animation-direction:' - allows us to control how our animation plays out. The values can be:

    • normal (default): the animation plays from start to finish.

    • reverse: the animation plays from finish to start.

    • alternate: the animation plays from start to finish and back to start.

    • alternate reverse: the opposite of 'alternate'


  • 'animation-fill-mode:' - allows us to choose what styles we want to apply to our element/selector after the animation has finished. Normally animations don't have an effect on an element before or after the animation has occurred, but this property can override that! This property is only really used if the iteration-count property is set to a number (meaning the animation at one point finishes). The values can be:

    • none (default): no styles are applied before or after the animation occurs.

    • forwards: when the element has finished animating, it will remain as it looks at the end of the animation (i.e. the style values set by the last keyframe).

    • backwards: when the element has finished animating, it will remain as it looks at the beginning of the animation (i.e. the style values set by the first keyframe).

    • both: when the element has finished animating it gets the style of wherever it finishes. Usually applied when we are using both the animation-direction and animation-iteration-count properties.

  • 'animation-play-state:' - allows us to play and pause the animation. The default value is 'running' and if we set it to 'paused', it will pause the animation. This property is most effective when elements/selectors are combined with pseudo-classes. Resuming a paused animation will start the animation from where it left off at the time it was paused, rather than starting over from the beginning of the animation sequence.

  • 'animation:' - the shorthand property which specifies the above eight properties together, in the above order. For example, the following in the shorthand form is the same as writing in the longhand form:

    “transition: nameExample 2000ms ease-in 200ms 5 reverse forwards running;”

    "animation-name: nameExample;
    animation-duration: 2000ms;
    animation-timing-function: linear;
    animation-delay: 200ms;
    animation-iteration-count: 5;
    animation-direction: reverse;
    animation-fill-mode: forwards;
    animation-play-state: running;"

    Note: if we don't specify a value in the shorthand property, the default value will be used. The default values for the shorthand property are as follows: "none 0 ease 0 1 normal none running". That is, if we don't set a duration time value here nothing will happen, because the default value is 0!

    Note 2: we can apply multiple animations within one 'animation:' property by separating them with commas.

CSS Animations: Transform (2D)

The CSS 'transform:' property is special property because it accepts multiple different types of values of 'methods' within the one property. 'transform:' accepts these four main methods:

  1. scale
  2. rotate
  3. translate
  4. skew

What's even more unique about the property is that within these methods are sub-methods, or sub-values, which this same property also accepts. For example, within these main four methods we have all these sub-methods:

  1. scale: scale( ), scaleX( ), scaleY( ), scale3d( ), scaleZ( )
  2. rotate: rotate( ), rotate3d( ), rotateX( ), rotateY( ), rotateZ( )
  3. translate: translate( ), translateX( ), translateY( ), translate3d( ), translateZ( )
  4. skew: skew( ), skewX, skewY( )

Transform is better for changing an element's height, width, or page position than the 'animation:' property because, unlike the 'animation:' property, this property doesn't affect other elements on our webpage. That is, an element can grow, shrink, move around our whole page and not have an impact on anything else!

This also means that our transformed elements will overlap other page elements if our values make them so, similar to 'absolute' positioned elements. However, a difference here is that the 'absolute' positioned elements are removed from the document flow and don't occupy any space on the webpage, whereas transformed elements do (in their original un-transformed position).

Also, it's very common to see the transform property used in conjunction with the animation properties and keyframes so that an element 'transforms' while in motion.

We will begin with 2D transformations.

SCALE

The scale method increases or decreases the size of the element.

We do this along either the X axis (left-right), the Y axis (up-down), or both. That is, we increase/decrease the width and height of the element.

Within the brackets we give numerical values like: 0.5, 1, 3 etc which represent a percentage of the original element's size. For example, 0.5 = 50% (half the size), 1 = 100% (the same size) and 2 = 200% (double the size).

To increase/decrease the element's size in 2D we have three properties:

  • scaleX( ): changes the element's size along the X axis.

  • scaleY( ): changes the element's size along the Y axis.

  • scale(X, Y): is the shorthand property for the above two properties. If we just want to alter one value and not the other, all we have to do is write '1' for the value we don't want changing.

The translate method moves an element from its current position (according to the values given for the X-axis and the Y-axis). That is, either left or right of its current position, or up or down from it current position, or both.

We can use any of the standard length measurement units here (pixels, em, rem, percentages etc), including negative values, the only difference here is that percentage values 'translate' the element in relation to itself, not to a parent element. That is, 100% = 100% the size of the element.

  • translateX( ): moves the element along the X axis.

  • translateY( ): moves the element along the Y axis.

  • translate(X, Y): moves the element along the X and Y axis. If we only specify the X value, the Y value is presumed to be 0.

SKEW

The skew method 'skews' an element. It's like the rotation method but instead of just moving the clock hands (as it were), we are spinning the clock itself around. It's difficult to explain through words and easier to understand looking at examples. In general the skew method is useful for making logos. And like for the rotate method, we use 'deg' (degrees) as the value within the brackets.

To skew an element in 2D we have three properties:

  • skewX( ): skews the element along the X axis by the given angle.

  • skewY( ): skews the element along the Y axis by the given angle.

  • skew(X, Y): skews the element along the X and Y axis by the given angles. If we don't want to alter a value we can simply write '0deg', and if we don't specify the Y value it is presumed to be 0.

MATRIX

We can combine multiple transformations by separating them with a space. For example:

3D transformations and how to create them

In order to make 2D transformations before we used the 'transform:' property, followed by certain methods and values. For 3D animations it's the same.

Before this property allowed us to scale, rotate, translate or skew our objects in 2D. Now we can also do these things in in 3D (except for 'skew') and again without affecting anything else on the page. In order to be able to do this, a new axis is introduced: the Z axis. Rotate: rotateX( ), rotateY( ), rotateZ( ), rotate3d(X, Y, Z, angle) → shorthand property

Translate: translateZ( ), translate3d(X, Y, Z) → shorthand property

However, in theory, we shouldn't be able to notice the 3D effect on these transformations yet. Why? Because for 3D effects with CSS we need perspective.

PERSPECTIVE

When we rotate an object in 2D, we can either rotate it clockwise or anti-clockwise on our screen using the 'rotate( ) method. When we use the 3D 'rotateX( )', 'rotateY( )' or 'rotateZ( )' methods we are asking our object to rotate around a particular axis (X, Y, Z). However, simply applying 3D rotation methods to an object won't make it magically appear in 3D on our screen. Instead, our 2D object will just appear smaller and/or more squished.

For example, in the video below we are using the 3D rotate methods on the green square. But notice that at first the square just looks like it's getting smaller and more squished, it doesn't look like it's in 3D but in 2D. It's only after we add 'perspective' that the square suddenly starts to look like it's 3D. For the Z axis, nothing changes.

In order to properly see and appreciate the 3D rotation, or indeed any 3D transformation on our webpage, we need something called perspective.

Before perspective we're just working with a 2D, flat screen, which has an X axis and a Y axis. However, applying the 'perspective:' property introduces the Z axis into our screen. That is, we're not dealing with a flat, 2D screen anymore. We have to imagine that the screen suddenly has depth. We have to imagine that our screen extends out towards us. That we can look into it and perceive depth just like when we're in the real world looking across a landscape and towards the horizon. (See the above image showing the three axes again to help visualise this).

In the 3D world - if we are looking at a landscape - things eventually get smaller and disappear the more distant they get, like looking at a road going into the horizon (see image below). This is because of something called the vanishing point - when things disappear from view the further they are from us.

So, by applying perspective, by applying a Z axis to our screen, suddenly the objects on our webpage suddenly also have depth and thus a vanishing point, producing a 3D effect.

In CSS, when we apply the 'perspective:' property we also have to choose where our perspective is located via the length unit value we give to our property. It can be located further away from the object by giving it a big length unit value (e.g. 500px), or closer to the object by giving it a small length unit value (50px). The different distances we set have interesting impacts on the appearance of the 3D object.

If we position ourselves close to the object (close to the starting point of the Z axis) the 3D effect is amplified. The part of the object which is closer to us will be quite wide, and the part of the object further away from us will be noticeably thinner/smaller. Note: a Z axis value of 0 means no 3D effect will be noticed.

If we position ourselves far away from the object (from the Z axis starting point) the 3D effect is diminished. The part of the object which is closer to us will be wider than the part of the object further away from us, however, the difference will be less noticeable because as we get further away from objects, in general the smaller they get and differences in size are less noticeable.

1) we have to combine perspective with our 3D transformations in order for them to work and have an effect (such as in the images above).

2) perspective introduces the Z axis to our CSS world, giving objects depth and a vanishing point, and determines how far our eyes are located from the object in our 3D CSS landscape.

Perspective Property vs Method

'perspective:' property = its own property which exists independently and can be applied directly to elements/selectors, typically the parent element to give multiple children the same perspective - but only direct descendant children. That is, applying this property to a parent/container creates a 3D-space in which all the transformed (direct descendant) children elements operate cohesively. The property takes length unit values.

Note: once we have applied perspective to the parent container, it's only then the transformation that we apply to the children elements behaves in a 3D fashion. That is, the transformations of children elements always behave according to the perspective rules of the parent element.

'perspective( )' method = is a value set for the 'transform:' property that is applied to individual child elements being transformed, in order to give it its own perspective. This function, like the 'perspective:'' property, also sets a distance between the user and the starting point of the Z axis, but just for this individual element. If used on multiple elements the transformed elements don't line up together because each element has its own perspective, its own vanishing point. It is used in conjunction with the functions rotateX( ), rotate( ), and rotate( ) which always follow the perspective( ) function in order for the effect to work. The function takes length unit values.

A note about perspective and transformations

We can “chain” multiple transform methods to the 'transform:' property. But when we do this, there are three very important things to consider:

  1. When rotating an object, its coordinate system is transformed along with the object.

  2. When translating (moving) an object, it moves relative to its own coordinate system (rather than its parent's coordinates).

  3. The order in which these values are written can (and will) change the end result.

To underscore how important the order of values is to the transform property, let's take a look at a couple of quick examples. First, a simple two-dimensional (2D) transformation of two squares that both have the same transform values, but declared in a different order:

…will have the same effect as:

That's because in the first line we moved the object on the X axis before rotating it, but in the second line we rotated the object, changed its coordinates, then moved it on the Z axis. Same result, different values.

Frames of reference

Note that transformations all operate with respect to a frame of reference. To best understand this concept, visualize the following scenario:

A train is moving along a railroad. This train is moving north, relative to the Earth.

However, relative to the Earth, the apple is actually moving north and up and down.

Other 3D related properties

Now we've understood the relationship between the 'transform:' propery and perspective, as well as the 'perspective-origin:' property, we can now look at the other 3D related transform properties:

'transform-origin:' property:
we saw before that this property allows us to define the origin for whatever transformations we have specified in the 'transform:' property and that it can only come after the original 'transform:' property has been applied.

Before we only changed the X and Y axis values. Now we can change the Z axis value, allowing us to move the object forwards or backwards in the 3D landscape, and only accepting length unit values. The Z axis value comes after the X and Y axis values (e.g. the default value is 50% 50% 0).

We can visualise and play around with the effect of the Z axis here if we turn the red box on its side and then change the Z axis value. You will notice it moves back and forth.

'transform-style:' property:
by giving this property the value 'preserve-3d' we ensure that our children elements inherit the 'perspective' given to their parents which enables the 3D landscape to exist, and thus are rendered within the same 3D space as their parents. This subsequently means that they are placed according to their position in 3D space not their order in the DOM. Can only come after the original 'transform:' property has been applied.

By default the value of this property is 'flat' so this property only has an effect when we want to do 3D transformations!

'backface-visibility:' property:
defines whether or not we can see the backface of an element (a mirror image of the front-face being displayed) when it's visible to the user. This property is useful when an element is rotated and allows us to choose what the user sees (or not).

The default value = 'visible'. The other possible value is 'hidden'.

Vendor Prefixes

CSS vendor prefixes (or simply 'CSS prefixes') are things we add into our CSS code in order to help make sure our CSS features work on as many browsers and devices as possible, i.e., to increase browser compatability.

This is not all the newer (or even regular) CSS features work on all browsers, especially on older browsers. All browsers have different ways of interpreting and rendering the CSS features and for this reason sometimes CSS prefixes are needed in order to make things work.

In addition to CSS prefixes, it's also good programming practice to combine our CSS prefix with a simpler, more standard fallback property in the case our newer CSS feature doesn't render. For example, setting just one color value instead of multiple values, having no animations instead of having them.

We typically want to apply our vendor prefixes last, after we have finished coding up our regular/standard CSS code. So when applying CSS prefixes we write our standard CSS first, for example: 

                        element {
                          animation: example 1s ease-out 0.5s infinite both
                        }

Then we write above this standard CSS property the prefix version. We write above the standard CSS property so that the standard version is the last line of code read for this element/selector and is the one the browser defaults to if the prefix versions above it fail. So now our code would look like this: 

                        element {
                          -prefix-animation: example 1s ease-out 0.5s infinite both
                          animation: example 1s ease-out 0.5s infinite both
                        }

Here is a list of the different CSS prefixes we can apply in our CSS:

  • -webkit- → Android, Chrome, IOS, Safari
  • -ms- → Internet Explorer
  • -mox- → Firefox
  • -o- → Opera

When adding CSS prefixes to our CSS code, the shorthand properties come into their own. This is because, looking at the 'animation:' property example from before, if we used each individual animation property (e.g. 'animation-duration:', 'animation-name:', 'animation-timing-function:' etc) then we would have to make a prefix copy for each individual property, which would make our CSS code look bloated.

This is basically all we need to know about vendor prefixes, but below is some very useful additional information that we can use to help us apply and effectively use our vendor prefixes.
